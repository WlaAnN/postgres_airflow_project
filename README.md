# postgres_airflow_project

Учебный ETL/ELT-проект, имитирующий аналитическую платформу подписочного сервиса: данные о пользователях, тарифах, промокодах и транзакциях проходят через слои **bronze → silver → gold** и оркестрируются с помощью **Apache Airflow**. Хранилище — **PostgreSQL**, администрирование — через **pgAdmin**. Всё поднимается одной командой через **Docker Compose**.

## Стек технологий

- **Apache Airflow 3.2.2** (CeleryExecutor) — оркестрация пайплайнов
- **PostgreSQL 16** — хранилище данных (отдельная база для Airflow и отдельная для проекта)
- **Redis** — брокер очередей для Celery
- **pgAdmin 4** — веб-интерфейс для просмотра БД
- **PL/pgSQL** — хранимые процедуры загрузки данных и SQL-тесты качества данных
- **Docker / Docker Compose** — запуск всей инфраструктуры
- **uv** — управление Python-зависимостями

## Архитектура данных

Проект построен по принципу медальонной архитектуры (Bronze / Silver / Gold):
- **Bronze** — «сырые» данные, как они приходят из источника (staging-таблицы без ограничений).
- **Silver** — нормализованная звёздная схема (Star Schema) с измерениями (`dim_users`, `dim_tariffs`, `dim_promocodes`, `dim_dates`) и фактовой таблицей `fact_transactions`. Используется подход **SCD Type 2** для версионирования тарифов и промокодов (`tariff_version`, `promocode_version` считаются через оконные функции по датам действия).
- **Gold** — аналитические витрины (view) для отчётности: сводка по использованию промокодов, разбивка по дням/неделям/месяцам/кварталам/годам, по тарифам и по промокодам.

Хранимые процедуры (`sql_scripts/procedures/`) переносят данные из bronze в silver: чистят дубликаты, нормализуют значения (например, пол: `Male`/`M`/`Female`/`F` → `M`/`F`), делают upsert (`INSERT ... ON CONFLICT DO UPDATE`) и логируют каждую загрузку в таблицу `elt_log` (количество строк, диапазон дат, статус, текст ошибки при сбое).

## DAG'и Airflow

| DAG | Расписание | Назначение |
|-----|------------|------------|
| `db_init_dag` | `@once` (один раз) | Инициализация БД: создаёт bronze/silver/gold слои, таблицу логов, заполняет справочник дат, переносит начальные данные из staging и добавляет комментарии к объектам БД (`docs.sql`) |
| `extract_data` | ежедневно в 16:30 | Имитирует выгрузку новых данных из источника в staging-таблицы (`insert_staging.sql`) |
| `clean_and_load_data` | ежедневно в 17:00 | Последовательно вызывает процедуры загрузки (`load_dim_users`, `load_dim_tariffs`, `load_dim_promocodes`, `load_fact_transactions`), затем прогоняет полный набор SQL-тестов (`run_all_tests.sql`) |

Расписание задано через `CronTriggerTimetable` с часовым поясом UTC+5 (Уфа).

## Тестирование качества данных

В `sql_scripts/tests/` реализованы три уровня проверок, которые запускаются скриптом `run_all_tests.sql`:

- **Level 1** — целостность silver-слоя: таблицы не пустые, корректность дат, математика скидок (`full_price - discount_amount = net_price`), целостность внешних ключей в `fact_transactions`.
- **Level 2** — согласованность между bronze и silver (совпадение количества строк, корректность флага `promocode_is_used`).
- **Level 3** — согласованность между silver и gold (суммы выручки в детальной витрине совпадают с агрегатами во всех data mart).

Каждый тест возвращает `0` (ОК) или `1` (найдена проблема), что удобно использовать для последующего алертинга.

```
.
├── airflow/
│   ├── config/airflow.cfg          # конфигурация Airflow
│   ├── dags/                       # DAGs

│   ├── tests/level_1|2|3/          # SQL-тесты качества данных
│   ├── insert_staging.sql          # генерация тестовых данных в bronze
│   ├── insert_all_dim_fact.sql     # начальное наполнение silver-слоя
│   ├── run_all_tests.sql           # запуск всех тестов одним скриптом (psql)
│   └── docs.sql                    # комментарии к таблицам/витринам
├── docker-compose.yaml             # вся инфраструктура: Airflow, Postgres x2, Redis, pgAdmin
├── .env.template                   # шаблон переменных окружения
├── pyproject.toml / uv.lock        # Python-зависимости
└── .python-version
```

## Как запустить

### Требования

- Docker и Docker Compose
- Не менее 4 ГБ RAM и 2 CPU, выделенных Docker (стандартное требование Airflow)

### 1. Настройка окружения

Скопируйте шаблон переменных окружения и при желании поменяйте пароли:

```bash
cp .env.template .env
```

На Linux также укажите свой `AIRFLOW_UID`, чтобы избежать проблем с правами на файлы:

```bash
echo "AIRFLOW_UID=$(id -u)" >> .env
```

Сгенерируйте `FERNET_KEY` (используется Airflow для шифрования соединений) и добавьте его в `.env`:

```bash
python3 -c "from cryptography.fernet import Fernet; print('FERNET_KEY=' + Fernet.generate_key().decode())" >> .env
```

(Опционально) Настройте автоподключение pgAdmin к БД проекта:

```bash
cp pgadmin/servers.json.template pgadmin/servers.json
# удалите из файла все строки-комментарии перед запуском
```

### 2. Запуск проекта

```bash
docker compose up -d
```

Поднимутся: Postgres для метаданных Airflow, Redis, Airflow (api-server, scheduler, dag-processor, worker, triggerer), отдельный Postgres для данных проекта (`my_postgres`) и pgAdmin.

### 3. Подключение Airflow к рабочей БД

В UI Airflow (см. ниже) создайте соединение `my_postgres_conn` (Admin → Connections) на базу `project_db`, либо используйте уже прописанное в `docker-compose.yaml` соединение `AIRFLOW_CONN_MY_POSTGRES_CONN`

!!!(Рекомендую использовать существующее соединение)!!!.

### 4. Запуск пайплайна

1. Откройте Airflow UI и проверьте запуск `db_init_dag` — он должен создать всю структуру БД и наполнить её начальными данными. Этот DAG должен автоматически запуститься после поднятия airflow. В случае если DAG не отработал успешно, запустите его вручную через какое-то время после выполнения docker compose. Такое может произойти, если база данных поднималась дольше, чем время прописанное в таймауте этого dag.
2. Включите DAG'и `extract_data` и `clean_and_load_data` — они будут выполняться по расписанию ежедневно (16:30 и 17:00 соответственно) и грузить новые тестовые данные.

### Веб-интерфейсы

| Сервис | URL | Логин/пароль по умолчанию |
|--------|-----|----------------------------|
| Airflow UI | http://localhost:8080 | `airflow` / `airflow` |
| pgAdmin | http://localhost:8081 | значения из `.env` (`MY_PGADMIN_DEFAULT_EMAIL` / `MY_PGADMIN_DEFAULT_PASSWORD`) |

## Что показывает этот проект

- Построение классической медальонной архитектуры данных (bronze/silver/gold) на чистом PostgreSQL без дополнительных ETL-фреймворков.
- Проектирование звёздной схемы с поддержкой версионирования измерений (SCD Type 2 подход для тарифов и промокодов).
- Оркестрацию многошаговых SQL-пайплайнов в Airflow с зависимостями между задачами, сенсорами и расписаниями на cron.
- Логирование процесса загрузки данных и обработку ошибок прямо в хранимых процедурах.
- Многоуровневое автоматизированное тестирование качества данных на SQL.
